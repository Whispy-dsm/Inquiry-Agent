# Agent Workflow Convention

## 분석 우선, 편집은 나중에

분석이 필요한 작업에서는 편집 전에 관련 파일, 프롬프트, 도구 계약, 평가 케이스를 먼저 읽고 변경 영향을 확인합니다.

적용 대상:

- 문의 처리 흐름 설계
- 프롬프트 수정
- 문의 분류 기준 변경
- 도구 호출 또는 외부 지식 소스 연동
- 응답 품질 개선
- 보안, 개인정보, 정책 검토
- 리팩토링 제안

## 문의 처리 기본 흐름

1. 문의 원문과 채널 정보를 수집합니다.
2. 문의 의도, 긴급도, 필요한 근거, 개인정보 포함 여부를 분류합니다.
3. 답변에 필요한 지식 소스나 도구를 확인합니다.
4. 확인된 근거로 응답 초안을 생성합니다.
5. 정책 위반, 근거 부족, 권한 부족, 고위험 문의는 사람에게 에스컬레이션합니다.
6. 처리 결과와 근거, 실패 사유를 추적 가능한 형태로 남깁니다.

## 2단계 프로세스

### Phase 1 - 분석 및 문서화

1. 관련된 모든 파일을 읽고 현재 구조와 변경 영향을 확인합니다.
2. 발견한 내용을 마크다운 문서나 작업 로그로 작성합니다.
3. 문서에는 파일 경로, 문제 또는 개선 사항, 심각도, 제안 수정 방법을 포함합니다.
4. 즉시 수정 요청이 아니라면 분석 결과를 먼저 제시합니다.

### Phase 2 - 수정 적용

1. 승인되었거나 사용자가 명시적으로 즉시 수정을 요청한 항목만 수정합니다.
2. 한 번에 하나씩 수정합니다.
3. 각 수정 후 관련 테스트나 평가 케이스를 실행합니다.
4. 문제가 생기면 즉시 원인을 확인하고 작업 로그에 남깁니다.

## 예외

아래 작업은 분석 문서 승인 없이 바로 진행할 수 있습니다.

- 사용자가 명시적으로 즉시 수정을 요청한 경우
- 오타, 링크, 문서명처럼 동작 영향이 없는 단순 수정
- 테스트나 평가 케이스 작성
- 문서나 에이전트 지침처럼 런타임 동작을 바꾸지 않는 단순 추가 작업

## 검증 기준

- 변경 범위에 가장 가까운 테스트나 평가 케이스를 먼저 실행합니다.
- 프롬프트, 분류 기준, 도구 계약, 지식 소스를 건드렸다면 회귀 평가 범위를 넓힙니다.
- 테스트를 실행하지 못한 경우에는 완료 보고에 이유와 남은 위험을 명시합니다.
