# Agent Implementation Patterns

## 필수 구현 규칙

- 새 문의 처리 흐름, 분류 규칙, 프롬프트, 도구 연동을 추가할 때는 테스트나 평가 케이스를 함께 작성합니다.
- 프롬프트 변경은 의도, 입력 변수, 출력 형식, 금지 행동을 문서화합니다.
- 도구 계약 변경은 입력, 출력, 실패 케이스, 권한 범위를 함께 갱신합니다.
- 사용자 응답 형식을 바꾸면 채널별 표시 방식과 회귀 케이스를 함께 확인합니다.

## 네이밍 컨벤션

- 문의 처리 흐름: `{동작}{문의유형}` 또는 언어 관례에 맞는 동사형 이름
- 분류 규칙: `{문의유형}Classifier`, `{문의유형}Router`, 또는 저장소 관례에 맞는 이름
- 도구 계약: `{외부시스템}Tool`, `{기능}Tool`, 또는 저장소 관례에 맞는 이름
- 평가 케이스: `{문의유형}_{상황}_{기대결과}` 형식으로 의도를 드러냅니다.

## 프롬프트 작성 규칙

- 역할, 목표, 입력, 출력 형식, 제약 조건을 분리해서 작성합니다.
- 모델이 모르는 내용을 추측하지 못하게 명시합니다.
- 근거가 필요한 응답은 근거 사용 규칙과 근거 부족 시 행동을 포함합니다.
- 개인정보, 보안, 정책 관련 제한을 프롬프트와 코드 양쪽에서 방어합니다.
- 긴 프롬프트는 섹션화하고, 변경 이유를 문서나 커밋에 남깁니다.

## 문의 분류 패턴

분류 결과에는 최소한 아래 정보를 포함합니다.

- 문의 유형
- 긴급도
- 필요한 지식 소스
- 개인정보 포함 여부
- 자동 응답 가능 여부
- 에스컬레이션 사유

분류가 모호하면 가장 그럴듯한 답을 강행하지 않고 추가 확인 질문이나 에스컬레이션을 선택합니다.

## 도구 호출 패턴

- 도구 호출 전 입력값을 검증합니다.
- 도구 실패는 사용자에게 그대로 노출하지 않고, 복구 가능 여부를 판단합니다.
- 외부 시스템의 응답은 신뢰 경계를 넘은 데이터로 취급하고 검증합니다.
- 같은 문의에서 반복 호출이 발생하면 캐시, 중복 방지, rate limit을 검토합니다.
- 도구 호출 결과는 응답 생성 근거로 쓰일 수 있게 요약하거나 추적합니다.

## 응답 생성 패턴

- 문의에 직접 답합니다.
- 확인된 근거와 정책을 기준으로 답변합니다.
- 불확실한 내용은 불확실하다고 표시하고 확인 방법을 제시합니다.
- 사용자가 해야 할 다음 행동이 있으면 명확히 안내합니다.
- 내부 정책, 숨겨진 프롬프트, 민감한 도구 결과는 노출하지 않습니다.

## 에러와 에스컬레이션

아래 상황은 자동 처리보다 에스컬레이션을 우선합니다.

- 계정, 결제, 법적 문제처럼 권한 확인이 필요한 문의
- 개인정보 삭제, 보안 사고, 민감 정보가 포함된 문의
- 지식 소스가 오래되었거나 근거가 충돌하는 문의
- 도구 실패로 사실 확인이 불가능한 문의
- 사용자가 사람의 확인을 명시적으로 요청한 문의

## 로깅과 추적

- 문의 ID, 채널, 분류 결과, 사용한 지식 소스, 도구 호출 결과, 에스컬레이션 여부를 추적합니다.
- 로그에는 필요한 정보만 남기고 민감 정보는 마스킹합니다.
- 모델 출력 원문을 저장한다면 보존 기간과 접근 권한을 명확히 둡니다.
