# Agent Architecture Convention

## 기술 스택 확인

작업 전 저장소의 실제 설정 파일을 확인합니다.

- 런타임과 언어
- LLM 또는 에이전트 실행 방식
- 문의 입력 채널
- 도구 호출 방식
- 지식 소스와 검색 방식
- 상태, 메모리, 로그 저장 방식
- 평가와 모니터링 도구

확인되지 않은 기술 스택을 전제로 코드를 작성하지 않습니다.

## 권장 구성

에이전트 구성 요소는 아래 역할이 드러나게 배치합니다. 실제 디렉터리 이름은 저장소의 기존 구조를 우선합니다.

```text
agent/
├── channels/        # 문의 유입 채널 어댑터
├── workflows/       # 접수, 분류, 근거 조회, 응답, 에스컬레이션 흐름
├── prompts/         # 시스템, 라우팅, 응답 생성 프롬프트
├── tools/           # 외부 도구 계약과 구현체
├── knowledge/       # 지식 소스 인덱스와 검색 정책
├── memory/          # 상태와 장기 기억 정책
├── policies/        # 개인정보, 보안, 에스컬레이션 정책
├── evals/           # 평가 데이터와 회귀 시나리오
└── observability/   # 로그, 트레이스, 품질 지표
```

## 핵심 규칙

- 문의 처리 흐름은 접수, 분류, 근거 조회, 응답 생성, 에스컬레이션 단계를 분리합니다.
- 프롬프트는 코드에 흩뿌리지 않고 추적 가능한 위치에 둡니다.
- 도구 호출은 입력, 출력, 실패 처리, 권한 범위를 명확히 정의합니다.
- 에이전트가 생성한 답변은 가능한 한 근거와 함께 추적 가능해야 합니다.
- 고위험 문의, 근거 부족 문의, 권한 밖 문의는 자동 답변보다 에스컬레이션을 우선합니다.
- 상태와 메모리는 목적, 보존 기간, 삭제 기준을 문서화합니다.
- 채널별 표현 방식은 분리하되, 핵심 문의 처리 판단은 공통 흐름에서 재사용합니다.

## 새 문의 처리 기능 추가 순서

1. 처리할 문의 유형과 성공 기준을 정의합니다.
2. 분류 기준과 에스컬레이션 기준을 작성합니다.
3. 필요한 지식 소스와 도구를 확인합니다.
4. 프롬프트 또는 workflow 단위를 추가합니다.
5. 정상, 경계, 실패, 에스컬레이션 평가 케이스를 작성합니다.
6. 로그와 추적 필드가 충분한지 확인합니다.
7. 관련 문서를 갱신합니다.
